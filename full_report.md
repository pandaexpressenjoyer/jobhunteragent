│  ### The Question (as it would likely be asked):                                                                                                                                                               │
│  > "Walk me through how you would take a piece of bare-metal, polling-based firmware — say, something reading a sensor over I2C in a `while(1)` loop — and redesign it to run under an RTOS like FreeRTOS.     │
│  What tasks would you create, what synchronization primitives would you use, and what specific bugs or race conditions would you need to guard against that didn't exist in the bare-metal version?"           │
│                                                                                                                                                                                                                │
│  *(Likely follow-up): "What's the difference between a mutex and a binary semaphore, and when would using the wrong one bite you in production?"*                                                              │
│                                                                                                                                                                                                                │
│  ### Ideal Engineering Response Strategy:                                                                                                                                                                      │
│                                                                                                                                                                                                                │
│  **Structure the answer in 3 layers — Architecture → Primitives → Failure Modes:**                                                                                                                             │
│                                                                                                                                                                                                                │
│  1. **Architecture (task decomposition):**                                                                                                                                                                     │
│     - Identify natural concurrency boundaries: e.g., a `SensorReadTask` (owns the I2C bus, blocks on a timer or interrupt), a `ProcessingTask` (consumes data via queue), and a `CommsTask` (handles           │
│  UART/output).                                                                                                                                                                                                 │
│     - Explicitly state priority assignment logic: time-sensitive I/O tasks get higher priority than processing/logging tasks, and justify *why* (priority inversion risk, deadline criticality).               │
│                                                                                                                                                                                                                │
│  2. **Primitives (be precise, don't hand-wave):**                                                                                                                                                              │
│     - Use a **queue** to move sensor data from `SensorReadTask` to `ProcessingTask` — this is inherently thread-safe and avoids needing a mutex for that handoff.                                              │
│     - Use a **mutex** only when protecting a *shared resource* like the I2C bus itself if multiple tasks need to issue transactions on it (mutexes carry priority inheritance in FreeRTOS — semaphores         │
│  don't).                                                                                                                                                                                                       │
│     - Use a **binary semaphore** for ISR-to-task signaling (e.g., a DMA-complete interrupt waking a task) — NOT a mutex, because semaphores have no ownership concept and are safe to give from an ISR         │
│  context (`xSemaphoreGiveFromISR`).                                                                                                                                                                            │
│                                                                                                                                                                                                                │
│  3. **Failure modes (this is what separates a strong candidate from a memorized-definitions candidate):**                                                                                                      │
│     - **Priority inversion**: low-priority task holds the I2C mutex, high-priority task blocks waiting for it, medium-priority task preempts the low-priority one — classic Mars Pathfinder bug. Solution:     │
│  FreeRTOS mutexes have built-in priority inheritance; semaphores don't, so choosing the wrong primitive silently reintroduces this bug.                                                                        │
│     - **Stack sizing per task** — bare-metal had one big stack; now each task needs its own sized stack, and undersizing causes silent heap corruption that's brutal to debug.                                 │
│     - **ISR latency changes**: with an RTOS, an ISR now has to be *short* and defer real work to a task via a semaphore/queue-from-ISR — a bare-metal ISR often just did the work inline.                      │
│                                                                                                                                                                                                                │

│  ## 🎯 QUESTION 2: Hardware Protocol Debugging (SPI/I2C/UART)                                                                                                                                                  │
│  **(Direct hard requirement: Block — "SPI, I2C, UART, USB"; implied: Geotab — "understanding of electronics influencing firmware development/debugging")**                                                     │

│  ### The Question (as it would likely be asked):                                                                                                                                                               │
│  > "You've wired up a sensor over SPI and it's returning garbage or all-zero data — walk me through your debugging process step by step. What would you check first, what tools would you reach for, and how   │
│  would your approach differ if the same problem happened on I2C instead?"                                                                                                                                      │
│                                                                                                                                                                                                                │
│  *(Likely follow-up): "What's actually different at the electrical/protocol level between SPI and I2C that would change how you'd debug each one?"*                                                            │
│                                                                                                                                                                                                                │
│  ### Ideal Engineering Response Strategy:                                                                                                                                                                      │
│                                                                                                                                                                                                                │
│  **Structure as a debugging funnel — from cheapest/fastest checks to instrumentation:**                                                                                                                        │
│                                                                                                                                                                                                                │
│  1. **Configuration-level checks first (cheap, no tools needed):**                                                                                                                                             │
│     - Confirm **SPI mode** (CPOL/CPHA — mode 0/1/2/3) matches the datasheet; this is the #1 cause of garbage SPI data for a first-time integration.                                                            │
│     - Confirm **clock speed** isn't exceeding the peripheral's rated max (common intern mistake — running SPI at MCU max speed without checking the sensor's datasheet limit).                                 │
│     - Confirm **chip-select (CS) polarity and timing** — active-low vs active-high, and whether CS is toggled correctly around each transaction.                                                               │
│     - For I2C: confirm **pull-up resistors** are present and appropriately sized (I2C is open-drain — without pull-ups, the bus simply won't work, unlike SPI which is push-pull and doesn't need them).       │
│                                                                                                                                                                                                                │
│  2. **Instrumentation (this is where the logic analyzer answer needs to live):**                                                                                                                               │
│     - "I'd hook up a logic analyzer across MOSI/MISO/SCLK/CS and visually confirm the bit pattern being sent matches the command byte from the datasheet, and that the slave is actually driving MISO during   │
│  the read window."                                                                                                                                                                                             │
│     - For I2C specifically: check for a **NACK** on the address byte (wrong 7-bit address, or address shifted by 1 bit — extremely common bug from R/W bit confusion) — this is visible immediately on a       │
│  logic analyzer trace.                                                                                                                                                                                         │
│     - Mention using an **oscilloscope** to check signal integrity if the logic analyzer shows correct logic-level timing but data is still wrong — could indicate a rise-time/capacitance issue at higher      │
│  clock speeds.                                                                                                                                                                                                 │
│                                                                                                                                                                                                                │
│  3. **Protocol-level conceptual differentiation (this is the "do you actually understand it" test):**                                                                                                          │
│     - SPI: full-duplex, push-pull, no addressing (CS does the addressing), faster, more pins.                                                                                                                  │
│     - I2C: half-duplex, open-drain + pull-ups, addressed bus (multi-drop), slower, fewer pins, ACK/NACK built into the protocol itself — which means I2C debugging often centers on the ACK bit, while SPI     │
│  debugging centers on clock mode/timing.                                                                                                                                                                       │
│                                                                                                                                                                                                                │
│  ### 🔍 Evaluator Talking Points / Red Flags:                                                                                                                                                                  │
│  - ✅ **Green flag:** Candidate checks CPOL/CPHA mode *before* reaching for a logic analyzer — shows debugging efficiency, not just tool-reliance.                                                             │
│  - ✅ **Green flag:** Candidate explicitly connects pull-up resistor absence to I2C's open-drain design — shows electrical understanding, not just software-level protocol memorization.                       │
│  - 🚩 **Red flag:** Candidate jumps straight to "I'd use a logic analyzer" without checking configuration first — real engineers check the cheap stuff first.                                                  │
│  - 🚩 **Red flag:** Candidate can't articulate why I2C needs pull-ups and SPI doesn't.                                                                                                                         │
│  - **What this maps to in the pipeline:** Aditya has verified I2C/UART experience from his MSP432/ESP32 work but no SPI — this question format lets him bridge the gap by reasoning from first principles      │
│  (I2C experience → extrapolate to SPI) rather than needing direct SPI project experience, which is a legitimate and often-praised interview strategy for a 2nd-year student.                                   │
